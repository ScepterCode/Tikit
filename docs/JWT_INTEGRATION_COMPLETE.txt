TOKEN FLOW:
===========
1. User logs in with email/password on frontend
2. Frontend calls Supabase auth API
3. Supabase returns JWT token
4. Frontend stores token in session
5. Frontend sends token in Authorization header for API calls
6. Backend validates token and extracts user info
7. Backend processes request with user context

TOKEN STRUCTURE:
================
Supabase JWT tokens contain:
  - iss: Issuer (Supabase URL)
  - sub: User ID
  - email: User email
  - role: "authenticated" (Supabase role)
  - user_metadata: Custom user data
    - first_name
    - last_name
    - role (custom: admin, organizer, attendee)
    - state
    - organization_name (if organizer)
    - organization_type (if organizer)

BACKEND ENDPOINTS:
==================
GET /api/auth/me
  - Validates JWT token from Authorization header
  - Returns user information
  - Falls back to mock token if JWT validation fails
  - Requires: Authorization: Bearer <token>

All other endpoints:
  - Can use get_user_from_request() helper to get user info
  - Automatically validates JWT or mock token
  - Maintains role-based access control

TROUBLESHOOTING:
================
Issue: "Token not issued by Supabase"
Solution: Check token issuer format (should contain "supabase")

Issue: "Token has expired"
Solution: Get new token from Supabase (tokens expire after 1 hour)

Issue: "Missing or invalid authentication token"
Solution: Ensure Authorization header is present and properly formatted

Issue: Backend returns 401 Unauthorized
Solution: Check that JWT token is valid and not expired
